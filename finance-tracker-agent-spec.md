# Personal Finance Tracker — Agent Build Spec
## Local-First, Tauri 2 (Desktop + Mobile), Ollama-Assisted Insights

This document is written to be handed directly to an AI coding agent
(Claude Code, Cursor, etc.) in stages. Work through it milestone by
milestone — do not ask the agent to build the whole app in one shot.
Sections marked **"Build yourself"** should not be delegated to the agent;
they're the parts that matter most in interviews.

---

## 1. Product Summary

A local-first personal finance app. All data lives in an encrypted SQLite
database on the user's device — no server, no cloud database, no account
system beyond a single local lock screen. The app tracks income and
expenses by category, lets the user set financial goals, and offers AI-
generated spending insights and advice (via Ollama on desktop, with a
rules-based fallback everywhere else), with a persistent disclaimer that
this is not professional financial advice.

**Explicitly out of scope, by design:** live bank account syncing (Plaid/
MX/Finicity-style). This would require a hosted backend to broker OAuth
tokens, which conflicts with the app's core "not hosted online" design.
Document this decision in the README rather than attempting a partial
implementation.

## 2. Tech Stack

| Layer | Choice |
|---|---|
| App shell | Tauri 2 (desktop: Windows/macOS/Linux; mobile: Android, iOS if a Mac is available) |
| Frontend | React + TypeScript + Vite |
| Styling | Tailwind CSS, responsive breakpoints for desktop vs. mobile layouts |
| Local database | SQLite via `tauri-plugin-sql`, encrypted with SQLCipher |
| Key derivation | Argon2id (derive DB encryption key from the user's lock-screen password) |
| Desktop AI | Ollama, called via local HTTP API (`http://localhost:11434`) from the Rust core |
| Mobile/fallback AI | Rules-based insight engine (no LLM) — see Section 6 |
| File import | CSV/OFX/QFX parser for one-time/on-demand bank statement import |
| Charts | Recharts |
| Packaging | Tauri CLI (`tauri build`, `tauri android build`, `tauri ios build`) |

## 3. Data Model

Build this schema by hand before prompting the agent for anything else.

```sql
-- Single local user / lock screen
CREATE TABLE app_lock (
  id INTEGER PRIMARY KEY CHECK (id = 1), -- enforce single row
  password_hash TEXT NOT NULL,           -- Argon2id hash, for login check
  kdf_salt TEXT NOT NULL,                -- salt used to derive the DB encryption key
  created_at TEXT NOT NULL
);

CREATE TABLE categories (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  type TEXT NOT NULL CHECK (type IN ('income', 'expense')),
  color TEXT
);

CREATE TABLE transactions (
  id INTEGER PRIMARY KEY,
  category_id INTEGER REFERENCES categories(id),
  amount REAL NOT NULL,               -- positive; sign implied by category type
  description TEXT,
  date TEXT NOT NULL,
  source TEXT NOT NULL DEFAULT 'manual' CHECK (source IN ('manual', 'imported')),
  import_batch_id INTEGER REFERENCES import_batches(id), -- null if manual
  created_at TEXT NOT NULL
);

CREATE TABLE import_batches (
  id INTEGER PRIMARY KEY,
  file_name TEXT NOT NULL,
  imported_at TEXT NOT NULL,
  transaction_count INTEGER NOT NULL
);

CREATE TABLE goals (
  id INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  target_amount REAL,
  target_date TEXT,
  category_id INTEGER REFERENCES categories(id), -- optional link, e.g. "reduce Dining"
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'completed', 'abandoned')),
  created_at TEXT NOT NULL
);

CREATE TABLE ai_insights_log (
  id INTEGER PRIMARY KEY,
  source TEXT NOT NULL CHECK (source IN ('ollama', 'rules_engine')),
  prompt_context TEXT,      -- what was sent, for your own debugging, not shown to user
  response_text TEXT NOT NULL,
  created_at TEXT NOT NULL
);
```

Note the `import_batch_id` on transactions: this lets you show "these 40
transactions came from your March statement import" and, if needed, undo
an entire import in one action.

## 4. App Structure — Tabs

- **Home tab**
  - Income vs. expenses summary (current month, with a prior-month comparison)
  - Breakdown by category (chart + list)
  - "Import Transactions" action (see Section 5)
  - A compact AI insights card — 1–2 auto-generated observations, with a
    "See more in AI tab" link, and the disclaimer visible whenever an
    AI-generated insight is shown, not just once per session
- **AI tab**
  - Chat-style interface for asking about spending habits ("Why did I
    spend more this month?")
  - Advice section — clearly separated from goal tracking, with a
    persistent, non-dismissable disclaimer banner:
    *"This is general information generated by an AI model, not financial
    advice from a licensed professional."*
  - Goals list — create/edit/view progress; goals can optionally be passed
    as context to the AI for more relevant advice, with the same disclaimer

## 5. Bank Statement Import (User-Triggered, One-Time or On-Demand)

This is **not** live sync — it's a file import the user triggers manually
whenever they want to pull in a new statement.

**Flow:**
1. User clicks "Import Transactions" and selects a CSV/OFX/QFX file
   exported from their bank's website.
2. Parse the file (support CSV first; OFX/QFX as a stretch — they're
   structured formats with existing Rust/JS parsing libraries).
3. Run each transaction through a **simple rule-based auto-categorizer**
   (keyword matching against the description — e.g., "STARBUCKS" → Dining,
   "SHELL OIL" → Transportation). Maintain a small, user-editable keyword-
   to-category mapping table so this improves over time.
4. **Show a review screen before committing anything to the database** —
   list all parsed transactions with their guessed category, let the user
   fix any miscategorized ones, then confirm the import as a batch.
5. Store the batch via `import_batches` + linked `transactions` rows.

**Agent prompt for this milestone:**
> Implement a CSV parser for bank transaction exports with columns for
> date, description, and amount (handle a few common bank export column
> naming variations). Add a keyword-based auto-categorization function
> that checks transaction descriptions against a `keyword_category_map`
> table. Build a review UI that shows all parsed transactions with editable
> category dropdowns before any database write happens. Only commit to the
> `transactions` table after user confirmation.

## 6. AI Insights — Two Backends Behind One Interface

Build a single Rust-side interface, e.g. `generate_insight(context) -> String`,
with two implementations selected by platform/availability:

**Desktop — Ollama backend:**
- Check if Ollama is running locally (`GET http://localhost:11434/api/tags`)
- If available, send a prompt built from the user's recent transaction
  summary + any active goals, asking for a short, specific observation or
  suggestion
- If Ollama isn't installed/running, fall back to the rules engine and
  tell the user why (e.g., "Ollama not detected — showing rule-based
  insights. Install Ollama for AI-generated insights.")

**Everywhere (mobile, or desktop fallback) — Rules engine:**
- No LLM at all — just pre-written templates filled in with real numbers:
  - "You spent {X}% more on {category} this month compared to last month."
  - "Your {goal_name} goal is {X}% complete, on pace to finish by {date}."
  - "{category} is your largest expense category this month at {amount}."
- This should feel genuinely useful on its own, not like a placeholder —
  spend real time on the rule conditions and thresholds, since this is
  what most users will actually see, especially on mobile.

**Important:** every single output from either backend — Ollama or rules
engine — must be labeled with its source and carry the disclaimer if it's
in the "advice" category (as opposed to a purely factual observation like
"you spent $340 on Dining this month," which doesn't need a disclaimer,
vs. "consider reducing Dining spending," which does).

**Agent prompt for this milestone:**
> Implement a Rust module with a single `generate_insight` function that
> takes a transaction summary and goal list as input. First check if
> Ollama is reachable at localhost:11434; if so, build a prompt from the
> input data and call Ollama's `/api/generate` endpoint. If unreachable,
> fall back to a rules-based template engine using the conditions I'll
> specify. Return a struct that includes the response text, which backend
> produced it, and whether it should be flagged as "advice" (needs
> disclaimer) vs. "observation" (factual, no disclaimer needed).

## 7. Local Lock Screen (Single User)

- On first launch: prompt to set a password. Hash it with Argon2id, store
  the hash in `app_lock`. Derive a separate encryption key from the same
  password (different salt) and use it to open/create the SQLCipher-
  encrypted database.
- On subsequent launches: prompt for password, verify against the hash,
  derive the same key, open the encrypted DB.
- No usernames, no multi-user support, no password reset flow beyond
  "delete local data and start over" (make this very clear in the UI —
  losing the password means losing the data, by design, since there's no
  server-side recovery).
- **Note for later, if you revisit cloud sync:** this single-password-
  derived-key model would need to change if you ever add multi-device
  sync, since you'd need a way to recover/rotate keys across devices
  without weakening local security. Worth a line in your README as a
  known limitation/future direction, not something to solve now.

**Build this yourself** — don't delegate the password hashing, key
derivation, or DB unlock logic to the agent without reviewing every line.
This is the part of the app most worth being able to explain in detail.

## 8. Milestones

1. **Planning** (by hand): finalize schema above, sketch Home + AI tab
   layouts for desktop and mobile
2. **Scaffolding**: Tauri 2 + React + Vite + `tauri-plugin-sql`, Android
   target initialized
3. **Lock screen + encrypted DB** (build yourself): Argon2id hashing, key
   derivation, SQLCipher setup
4. **Manual income/expense CRUD**: categories, transactions, basic Home
   tab dashboard
5. **Goals CRUD**: create/edit/track progress
6. **Bank statement import**: CSV parser, rule-based categorizer, review
   screen, batch commit
7. **AI insights — rules engine first**: get this working and genuinely
   useful before touching Ollama
8. **AI insights — Ollama backend**: desktop-only integration, with clean
   fallback to rules engine
9. **Responsive polish**: test on desktop window + Android emulator/device
10. **Testing**: unit tests for parsers, categorizer, insight generation
    logic (both backends), lock screen logic
11. **Packaging + docs**: builds for desktop + Android, README with
    architecture, schema, and the three scope decisions documented
    (single-user lock screen, Ollama-desktop-only, file-import-not-live-sync)
    with your reasoning for each

## 9. Interview Talking Points This Project Now Supports

- Local-first architecture and the specific privacy tradeoff it represents
- Encryption key derivation from a password vs. storing a static key
  (and what that does/doesn't protect against — e.g., it protects data at
  rest if the device is stolen, but the "forgot password = lose data"
  tradeoff is a deliberate design choice worth explaining)
- Why you chose file-based import over a hosted bank-sync API, and what
  it would take to add one later
- A platform-abstracted AI backend (Ollama vs. rules engine) and the
  reasoning for keeping mobile LLM-free rather than forcing an on-device
  model in v1
- Responsible AI UX — the observation-vs-advice distinction and why it
  matters for disclaimers
